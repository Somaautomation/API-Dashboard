## 🎯 Demo Session Agenda (2.5 Hours)

### Part 1: Architecture & Strategy Walkthrough (20 mins)

**Slide deck topics:**
1. **Target Architecture**
   - Single VM + Docker Compose (cost-effective for internal use)
   - Nginx reverse proxy with TLS termination
   - PostgreSQL + Redis for persistence/caching
   - Celery workers for async tasks

2. **Security Posture**
   - VPN-only or allowlist access
   - Basic auth layer on top of Nginx
   - TLS encryption (Let's Encrypt or company cert)
   - No public internet exposure

3. **Cost Optimization**
   - Single mid-tier VM (Hetzner CX22, AWS t3.small, ~$5-15/mo)
   - All services co-located (app, DB, Redis, worker)
   - Horizontal scaling not required for internal use

4. **High-Level Deployment Flow**
   ```
   Local Dev → Git Push
            → GitHub Actions CI/CD
            → Build & Push Docker Images
            → SSH Deploy to Production VM
            → Docker Compose Pull & Restart
            → Health Checks + Smoke Tests
            → Monitoring & Logging
   ```

### Part 3: Production Configuration Deep Dive (60 mins)

**Walk through each configuration file and discuss team responsibilities.**

#### 3.1 Environment Configuration (.env.prod)
```bash
# Show the template
cat .env.prod.example

# Discuss required secrets (DO NOT fill in live):
# - APP_SECRET_KEY (generated by team)
# - POSTGRES_PASSWORD (generated by team)
# - BOOTSTRAP_ADMIN_PASSWORD (set by team)
# - API_KEYS for external services (if any)
```

**Team Task:**
- [ ] Create `.env.prod` with strong values (stored securely)
- [ ] Use AWS Secrets Manager / Vault / encrypted store
- [ ] Rotate passwords quarterly

#### 3.2 Docker Compose Production Setup
**File:** `deploy/internal/docker-compose.prod.yml`

Walk through:
- PostgreSQL configuration (persistent volumes, healthchecks)
- Redis setup (append-only mode, persistence)
- Backend service (env_file, depends_on, healthchecks)
- Worker service (Celery, same image as backend)
- Network isolation (app_net)
- Logging configuration (json-file, log rotation)

**Key Points:**
- All services restart automatically on failure
- Healthchecks prevent cascading failures
- Volumes persist data across container restarts
- No port exposure except through Nginx

#### 3.3 Nginx Configuration
**Files:** 
- `deploy/internal/nginx/nginx.conf` (global)
- `deploy/internal/nginx/conf.d/site.conf` (application)
- `deploy/internal/nginx/conf.d/allowlist.conf` (IP filtering)

Walk through:
```bash
# Show the structure
tree deploy/internal/nginx/

# Key components:
cat deploy/internal/nginx/nginx.conf          # Worker config, gzip, logging
cat deploy/internal/nginx/conf.d/site.conf    # SSL, upstream backends, rate limiting
cat deploy/internal/nginx/conf.d/allowlist.conf  # IP allowlist
```

**Features:**
- TLS termination (certificates auto-renewed)
- HTTP→HTTPS redirect
- Basic auth (htpasswd)
- IP allowlist enforcement
- Load balancing across backend replicas (if scaled)
- Security headers (CSP, X-Frame-Options, etc.)

**Team Task:**
- [ ] Review allowlist IPs (office public IPs, VPN CIDR)
- [ ] Decide on cert strategy (company cert vs Let's Encrypt)
- [ ] Set up htpasswd users if using basic auth

#### 3.4 Deployment Scripts
**Walk through each script:**

```bash
cd deploy/internal/scripts

# 1. IP Allowlist Generation
cat generate-allowlist.sh
# → Produces allowlist.conf from team-provided IP list

# 2. Nginx Config Rendering
cat render-nginx-config.sh
# → Populates site.conf template with INTERNAL_DOMAIN, cert paths

# 3. Let's Encrypt Provisioning (if applicable)
cat provision-letsencrypt.sh
# → Runs certbot ACME challenge, stores certs in /etc/letsencrypt

# 4. Company Cert Installation (if applicable)
cat install-company-cert.sh
# → Copies company cert/key to Nginx container volume

# 5. Deployment Main Script
cat deploy.sh
# → Orchestrates: pull code → build images → docker-compose pull/up → healthchecks

# 6. Rollback Script
cat rollback.sh
# → Reverts to previous image tags and restarts services

# 7. Post-Deploy Checks
cat post-deploy-checks.sh
# → Smoke tests: HTTP 200, redirect working, auth working
```

### Part 4: Hands-On Configuration Tasks (60 mins)

**Breakout activities: Assign tasks to team members**

#### Task Group A: IP Allowlist & Networking (15 mins)
**Assigned to:** Network/Security team member(s)

1. Gather team's external IPs (office, VPN egress)
   ```bash
   # On local machine
   curl https://ifconfig.me
   
   # Each team member runs this and reports
   ```

2. Create allowlist config:
   ```bash
   cat > /tmp/allowlist_ips.txt << EOF
   10.0.0.0/8           # Office VPN CIDR
   203.0.113.100/32     # Office public IP
   198.51.100.0/24      # Remote office CIDR
   EOF
   
   deploy/internal/scripts/generate-allowlist.sh /tmp/allowlist_ips.txt
   cat deploy/internal/nginx/conf.d/allowlist.conf
   ```

3. Plan firewall rules:
   - Inbound 443 from allowlist only
   - Inbound 80 for ACME (if Let's Encrypt)
   - All other ports blocked
   - Logging enabled

**Deliverable:** `deploy/internal/nginx/conf.d/allowlist.conf` ready for deployment

---

#### Task Group B: TLS Certificate Strategy (20 mins)
**Assigned to:** Infra/Security team member(s)

**Option 1: Let's Encrypt (Automated)**
```bash
# Test locally first
docker run -it --rm \
  -v /tmp/letsencrypt:/etc/letsencrypt \
  certbot/certbot \
  certonly --standalone \
  -d api-dashboard.internal.company \
  --email ops@company.com

# Then run provisioning script on production VM
deploy/internal/scripts/provision-letsencrypt.sh
```

**Option 2: Company Certificate (Manual)**
```bash
# Gather from security team:
# - server.crt (public cert)
# - server.key (private key)
# - ca-bundle.crt (intermediates, if any)

# Place in secure location, then:
deploy/internal/scripts/install-company-cert.sh \
  /secure/path/server.crt \
  /secure/path/server.key \
  /secure/path/ca-bundle.crt
```

**Verify TLS in Docker:**
```bash
# After cert setup
docker-compose exec nginx \
  openssl x509 -in /etc/nginx/certs/server.crt -text -noout | grep -A2 "Validity"
```

**Deliverable:** Decision document: cert strategy, renewal plan, expiry monitoring

#### Task Group C: Environment Secrets & Configuration (15 mins)
**Assigned to:** Platform/DevOps lead

1. Generate strong secrets:
   ```bash
   # APP_SECRET_KEY (FastAPI)
   python3 -c "import secrets; print(secrets.token_urlsafe(32))"
   
   # POSTGRES_PASSWORD
   openssl rand -base64 32
   
   # Encryption key (if used)
   python3 -c "from cryptography.fernet import Fernet; print(Fernet.generate_key().decode())"
   ```

2. Create `.env.prod`:
   ```bash
   cp .env.prod.example .env.prod
   
   # Edit with real secrets (use secure vault)
   nano .env.prod
   
   # Example entries:
   # APP_SECRET_KEY=<generated>
   # POSTGRES_PASSWORD=<generated>
   # BOOTSTRAP_ADMIN_PASSWORD=<strong-admin-pw>
   # INTERNAL_DOMAIN=api-dashboard.internal.company
   # ALLOWED_CIDRS=10.0.0.0/8,203.0.113.100/32
   # POSTGRES_DB=apidashboard_prod
   # REDIS_URL=redis://redis:6379/0
   ```

3. Set up secrets storage:
   - AWS Secrets Manager / Parameter Store
   - HashiCorp Vault
   - Encrypted Git repo (not recommended long-term)
   - 1Password / LastPass (team credentials)

**Deliverable:** Secure `.env.prod`, documented secret rotation process

#### Task Group D: VM Provisioning & Deployment Dry-Run (15 mins)
**Assigned to:** Infrastructure team member(s)

1. **VM Base Setup** (done once, can be pre-done or live if time allows):
   ```bash
   # SSH to new VM
   ssh ubuntu@<vm-ip>
   
   # Bootstrap
   sudo apt-get update && sudo apt-get upgrade -y
   sudo apt-get install -y \
     docker.io docker-compose-plugin git curl gettext-base htpasswd
   
   # Add user to docker group
   sudo usermod -aG docker ubuntu
   newgrp docker
   
   # Clone repo
   git clone https://github.com/yourorg/APIDashboard.git /opt/apidashboard
   cd /opt/apidashboard
   ```

2. **Prepare deployment directory:**
   ```bash
   mkdir -p /opt/apidashboard/secrets
   mkdir -p /opt/apidashboard/data/{postgres,redis,uploads}
   chmod 700 /opt/apidashboard/secrets
   ```

3. **Dry-run deployment (non-prod branch):**
   ```bash
   cd /opt/apidashboard/deploy/internal
   
   # Source env (or create temp for demo)
   export DEPLOY_TAG=latest
   export BACKEND_IMAGE=myregistry/backend
   export FRONTEND_IMAGE=myregistry/frontend
   
   # Show what would be deployed
   docker-compose -f docker-compose.prod.yml config
   
   # Actually deploy to test VM
   bash scripts/deploy.sh
   
   # Wait and check health
   sleep 30
   bash scripts/post-deploy-checks.sh
   ```

**Deliverable:** Deployment runbook validated, team confident in `deploy.sh`

## 🔍 Troubleshooting Reference

### Common Issues During Demo

#### Issue: Docker service won't start
```bash
# Check Docker daemon
sudo systemctl status docker
sudo systemctl restart docker

# Check disk space
df -h

# Check port conflicts
sudo netstat -tlnp | grep 3000  # Frontend
sudo netstat -tlnp | grep 8000  # Backend
sudo netstat -tlnp | grep 5432  # Postgres
```

#### Issue: Nginx can't reach backend
```bash
# Check network
docker network ls
docker network inspect app_net

# Check backend health
docker-compose exec backend curl http://localhost:8000/health
```

#### Issue: PostgreSQL not ready
```bash
# Check logs
docker-compose logs postgres

# Manually check
docker-compose exec postgres pg_isready -U postgres
```

#### Issue: Celery worker not processing
```bash
# Check logs
docker-compose logs worker

# Check Redis connection
docker-compose exec redis redis-cli ping
```

## 📞 Q&A Template

**Q: How many servers do we need?**
A: Start with 1 VM (all services co-located). Scale to 2-3 only if internal usage >200 concurrent users.

**Q: Can we scale the database?**
A: Yes, migrate to managed PostgreSQL later. For now, single instance is sufficient.

**Q: What if certificates expire?**
A: Let's Encrypt auto-renews. Company certs need monitoring—set reminder 60 days before expiry.

**Q: How do we handle data backups?**
A: Automated daily snapshots of `/var/lib/postgresql/data` volume to S3 / cloud storage.

**Q: Can we blue-green deploy?**
A: Yes, with 2 VMs. For now, 5-min maintenance window acceptable for internal tool.

**Q: Who has SSH access to production?**
A: Only on-call DevOps team. All changes via GitHub → GitHub Actions → automated deploy.
